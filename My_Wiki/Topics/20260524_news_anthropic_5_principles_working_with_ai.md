## 🎯 핵심 키워드: "복리 (Compound)"

> "AI와 일을 마치고 나면 결과물(코드, 문서, 분석, 의사결정)이 남고, 매번의 수정 피드백이 어딘가에 누적되어야 한다. 그래야 다음 작업에서는 더 적은 설명으로 더 좋은 결과를 받을 수 있다."

이 5가지 원칙이 가리키는 방향은 **AI를 단순한 도구로 쓰는 것이 아니라, 같이 일하는 동료처럼 키워나가는 시스템을 짜는 것**.

---

## 📜 5가지 원칙 상세

### 원칙 1: 맥락을 인프라처럼

AI가 잘 작동하려면 우리 일의 맥락에 접근할 수 있어야 합니다. 매번 설명하지 말고, AI가 직접 찾아갈 수 있게 정리해둡니다.

| 실천 | 설명 |
|---|---|
| **디렉토리 구조화** | 코드: `~/src`, 지식: `~/vault/projects/`, `notes/`, `kb/` — AI가 grep/glob으로 빠르게 찾을 수 있게 |
| **프로젝트별 INDEX.md** | 관련 문서·채널 링크 + 간단한 설명 "이 문서는 X에 관한 것이고, Y를 다룰 때 읽으면 좋다" |
| **신입사원 온보딩** | 새 세션마다 백지 시작. 프로젝트별 CLAUDE.md에 약어 사전, 프로젝트 코드네임, 팀원 구분 |

### 원칙 2: 취향을 설정처럼

작업 스타일을 매번 말로 설명하지 말고, 파일로 저장해둡니다.

| 실천 | 설명 |
|---|---|
| **CLAUDE.md를 행동 계약서로** | "확신 없으면 추측하지 말고 모르겠다고 말하기", "내 접근법에 문제가 있으면 직접 지적하기", "실패하면 재시도 전에 근본 원인 먼저 조사하기" |
| **일주일에 한 번 이상 → 스킬로** | `/polish`, `/write`, `/daily` 같은 스킬. 마크다운 파일. AI가 필요할 때 자동으로 불러옴 |
| **일단 해보고 스킬로** | 완벽하게 시작하는 게 아니라, 일단 작업한 뒤 그 과정을 스킬로 정리 → 다음 사용 때 피드백 |

### 원칙 3: 자율을 위한 검증

AI에게 더 많이 맡기려면, 그 결과를 빠르게 검증할 수 있는 구조가 먼저 있어야 합니다.

| 실천 | 설명 |
|---|---|
| **가능한 한 빨리, 싸게** | 검증 사다리: 아래(자동 포맷팅, 린트)는 싸고 빠르고, 위(테스트, LLM 리뷰)는 비싸고 판단 필요. 낮은 단계에서 잡아냄 |
| **AI 스스로 검증** | 숫자로 평가되면 AI가 직접 평가 돌려 최적화. 브라우저 렌더링이면 AI가 직접 화면 확인 |
| **긴 작업은 다른 AI가 감시** | 터미널 두 개: 한쪽은 작업 AI, 다른 한쪽은 검토 AI. 실행 차원 + 방향 차원 별 점검 |

### 원칙 4: 위임으로 규모 확장

한 줄씩 지시하는 게 아니라, 큰 작업을 통째로 맡겨야 합니다.

| 실천 | 설명 |
|---|---|
| **더 큰 단위로 위임** | "이 평가 묶음 전부에 대해 격리된 컨테이너 빌드 → 테스트 → 분석 → 신뢰구간 계산 → 리포트 생성 → 슬랙 발송" — 의도·제약조건·성공 기준 명확히 |
| **여러 세션 동시** | 평소에 3~6개 세션 동시. git worktree로 코드 저장소 분리 |
| **세션 상태 한눈에** | 작업 완료 시 알림 소리. 터미널 창 제목에 진행/완료 표시 |

### 원칙 5: 루프 닫기

매번의 작업이 다음 작업의 맥락이 되도록 만듭니다.

| 실천 | 설명 |
|---|---|
| **공개적으로 작업** | 공유 문서, 코드 저장소, 채널에 남기면 다른 사람도, AI도 그 맥락 가져다 씀 |
| **과거 대화 기록에서 설정 업데이트 찾기** | "이것도 확인해줘", "이거 빠뜨렸어" 같은 표현이 많으면 → CLAUDE.md나 스킬에 추가. 저자 과거 2,500개 입력 분석 기반 |
| **주기적으로 정리·가지치기** | 설정 파일이 쌓이면 서로 충돌. 한 가지 규칙은 한 곳에만, 안 쓰는 항목은 제거 |

## 💡 시사점

1. **AI를 "동료"로 대우**: 새로운 팀원이 들어왔을 때 자연스럽게 하는 일(문서 모아주기, 일하는 방식 알려주기, 검토·피드백, 점점 더 큰 일 맡기기)을 AI에게 똑같이 적용
2. **맥락=인프라**: 모든 지식과 설정을 파일로 관리하고 AI가 직접 접근 가능하게
3. **복리의 힘**: 각 작업의 결과가 다음 작업의 맥락이 되어, 시간이 갈수록 적은 노력으로 더 좋은 결과
4. **검증→자율→위임의 흐름**: 검증 구조 먼저 → 안심하고 위임 → 규모 확장 → 루프 닫아 복리
5. **실용적 접근**: 완벽하게 시작하지 말고, 일단 해보고 그 과정을 파일로 정리 → 피드백으로 다듬기
