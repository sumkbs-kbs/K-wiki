# 프롬프트 말고 하네스 — 안드레 카파시와 코덱스 워크플로우 6원칙 🎯

> **영상:** [바비랩스 — 프롬프트 말고 하네스](https://www.youtube.com/watch?v=Yt67kU3cZfQ) (6분 5초)

## 핵심 요약

카파시가 정리한 **Claude MD 카드의 6원칙**을 코덱스에 얹어 해석합니다. 핵심은 **"모델에게 무엇을 하라고 말하지 마 — 성공 기준을 두고 루프를 돌게 내버려 둬"** 입니다.

---

## 주요 내용

### 6가지 워크플로우 원칙 (Karpathy 카드)

1. **코딩 전에 생각하기** — CLI `/plan`으로 큰 일을 단계로 쪼개고 모호성 ↓. 프로젝트 규칙은 `CLAUDE.md`/`AGENTS.md`에 저장소 레벨로 강제
2. **멈춰서 검증하기** — `/review`가 핵심. 브랜치/커밋/미커밋/변경 4가지 모드로 자신의 코드 리뷰. Diff 스테이커스로 검증
3. **단순함이 우선** — 압축으로 컨텍스트 오염 방지. 모델의 자기 혼란을 끌고 다니지 않음
4. **필요한 곳만 건드리기** — 코덱스의 Worktree 격리. 다른 에이전트의 작업 무결성 보장
5. **목표 중심 실행** — `goal 모드` (2026-05-21 정식 기능). 나쁜 예: "이 프로젝트를 완성해 줘" / 좋은 예: "로그인 접근성 개선, 기존 디자인 유지, MPM 테스트+린트 통과면 끝"
6. **서브 에이전트로 병렬화** — 조사원1/디버깅1/테스트1/최적화1/검증1. 코덱스의 서브에이전트 + 워크트리로 자동화

### 하네스 설계의 핵심 통찰

> "모델의 문제는 모델 자체가 아니다. 모델은 묻지 않고 가정하고, 단순한 일을 과도하게 설계하고, 혼란을 숨기는 경향이 있다. 워크플로우의 규칙을 인코딩하면 그런 잘못이 크게 줄어든다."

## 핵심 인사이트

- **프론트엔드 개발자를 위한 도구** → **에이전트의 작업장**. 코덱스는 300만명/주가 사용, 600개 릴리스. 더 이상 '조용한 도구'가 아님
- **하네스 설계 vs 프롬프트 설계**. 같은 날 공개된 Shots (맥OS 커맨드+2번 → 스크린샷 → 코덱스 자동 인젝션), Remote Computer Use (맥 잠긴 상태 모바일 이어가기)
- **변화 방향**: 이전 *"이 함수 작성해"* → 이제 *"여기에 목표가 있어. 이 제약은 절대 지켜. 테스트 잘하고 검증 철저히. 올바르게 될 때까지 루프 반복"*
- **질문 자체의 변화**: "AI에게 어떤 프롬프트를 던질까?" → **"AI가 일할 작업장을 어떻게 설계할까?"**
- **만능 아님**: 길게 돌리면 토큰/로그 폭주. "작은 꿈보다 작은 계약 먼저"

## 연관성

- **Antigravity-K**: 코딩 에이전트 파이프라인 설계 — Karpathy의 원칙이 Antigravity-K 아키텍처(재귀적 자기개선, 품질 게이트,ReflectionAgent)에 직접 반영 가능
- **SmallCode**: 소형 LLM 하네스 설계의 실제 구현 — 컨텍스트 예산 관리가 원칙 1의 자동화
- **AgentMemory**: 4-Tier 메모리 — 원칙 5(실패 시 어디서 멈출지) + 원칙 6(병렬 검증)과 직접 연관
