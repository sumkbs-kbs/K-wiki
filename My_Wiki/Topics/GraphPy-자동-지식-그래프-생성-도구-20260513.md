# 그래피파이(GraphPy) - 자동 지식 그래프 생성 도구

## 핵심 요약

그래피파이(GraphPy)는 Andrej Karpathy의 "로우 폴더(Raw Folder)" 워크플로우에서 영감을 받아 만든 **자동 지식 그래프 생성 도구**입니다. 코드, 문서, 논문, 이미지, 영상 등 폴더에 넣기만 하면 자동으로 지식 그래프를 생성하며, **71.5배~5배의 토큰 절감 효과**와 **결정론적 분석**으로 정확도를 높입니다.

## 1. 그래피파이란?

그래피파이는 복잡한 코드, 문서, 이미지, 영상 등 다양한 데이터를 **시각적 지식 그래프**로 연결하여 이해를 돕는 도구입니다.

- **영감**: Andrej Karpathy의 "LM Knowledge Base" 아이디어 (X/Twitter 포스팅)
- **핵심 기능**: 폴더에 파일만 넣으면 → 자동으로 지식 그래프 생성
- **지원 형식**: 20개 이상의 언어 + PDF, 논문, 스크린샷, 화이트보드 사진, 영상
- **출력 형식**: HTML, JSON, Markdown, Obsidian Vault, SVG, GraphML, Neo4j Cypher

### 기존 도구 한계 vs 그래피파이 해결책

| 기존 도구의 한계 | 그래피파이 해결 |
|---|---|
| `grep`/`find`: 문자열 검색만, 구조적 관계 미해석 | 구조와 의미 **동시 분석** |
| IDE 정적 분석: 단일 프로젝트 내에서만 동작 | **크로스파일** 관계 자동 발견 |
| 정적 분석 도구: 코드만 분석 가능 | 문서, 논문, 이미지까지 **범주 초월 분석** |
| 인베딩 기반: 비결정적, 환각 가능성 | **결정론적 AST 기반**으로 정확도 UP |
| 매번 전체 분석 | ** Incremental 캐싱**으로 5배 빠르게 |

## 2. 핵심 아키텍처

### 2.1 이중 경로 추출 (Dual-Path Extraction)

```
┌─────────────────────────────────────────────┐
│              입력 폴더 스캔                   │
├──────────────────┬──────────────────────────┤
│  Pass A: 구조적  │  Pass B: 의미적 (LLM)     │
│  추출 (트리시터)  │  추출 (Claude/Antigravity)│
│  ███████████████  │  ░░░░░░░░░░░░░░░░░░░░░  │
│  100% 로컬, 무료  │  토큰 소모 (의미 분석용)  │
│  결정론적, 재현   │  비결정론적, 컨텍스트 인식 │
├──────────────────┴──────────────────────────┤
│         추출 결과 병합 → 지식 그래프          │
└─────────────────────────────────────────────┘
```

**Pass A (구조적 추출):**
- 트리티시터(trie-ci-tree) → 23개 언어 파싱
- **추상구문트리(AST)** 를 통한 코드 구조 분석
- 100% 로컬 실행, 비용 없음, 항상 동일한 결과
- 함수/클래스/임포드/상속 관계 등 추출

**Pass B (의미적 추출):**
- Claude/Antigravity/Codex 등 LLM 활용
- 개념·엔티티 인식, 인용·참조·설계 근거 추출
- 이미지: UI/스크린샷/차트/다이어그램 내용 분석
- 신뢰도 태그: `Extracted`(1.0), `Inferred`(0.6-0.9), `Ambiguous`(0.1-0.3)

### 2.2 9단계 전체 파이프라인

| 단계 | 모듈 | 설명 |
|---|---|---|
| 1 | Detect | 입력 폴더 스캔, 파일 카테고리별 분류, 민간 파일 자동 필터 |
| 2 | Extract | AST 구조 분석 + LLM 시맨틱 분석 (병렬) |
| 3 | Cache | 해시 기반 캐싱 → 변경 파일만 재처리 (80%+ 캐시 히트) |
| 4 | Build | 노드·엣지를 NetworkX 그래프로 조립, 중복 제거 |
| 5 | Analyze | 간 노드, 예상치 못한 연결, 커뮤니티 감지(Laydens) |
| 6 | Visualize | HTML/SVG/GraphML/Neo4j/Cypher 다양한 시각화 |
| 7 | Transcribe | 영상·오디오 → OpenAI Whisper로 텍스트 변환 |
| 8 | Query | BFS/DFS 기반 그래프 탐색, 자연어 쿼리 |
| 9 | Report | `graph_report.md` 생성 — 분석 결과 정리 |

### 2.3 핵심 구성 요소

```
graphpy/
├── detect.py        # 파일 감지·카테고리 분류
├── extract_graphpy.py  # AST 구조적 추출 (핵심 모듈)
├── build.py         # NetworkX 그래프 조립
├── analyze.py       # 간 노드·커뮤니티·추천 질문 분석
├── cli.py           # CLI 인터페이스
└── utils.py         # 유틸리티
```

## 3. 그래프 구조

### 노드(Node) — 엔티티

- **코드에서**: 함수, 클래스, 모듈 → 노드
- **문서에서**: 개념, 인물, 기술 용어 → 노드
- 각 노드: 고유 ID, 소스 파일 경로, 타입 정보

### 엣지(Edge) — 관계

|-edge 유형 | 설명 | 신뢰도 |
|---|---|---|
| Call | 함수 호출 관계 | Extracted (1.0) |
| Import | 모듈 임포트 관계 | Extracted (1.0) |
| Inherited | 클래스 상속 관계 | Extracted (1.0) |
| Uses | 변수·상수 참조 | Extracted (1.0) |
| Semantic | LLM 추론 관계 | Inferred (0.6-0.9) |
| Ambiguous | 불확실 관계 | Ambiguous (0.1-0.3) |

### GraphPy 자체 분석 결과

- **노드**: 1,394개
- **엣지**: 2,141개
- **커뮤니티**: 63개 (Laydens 알고리즘)
- **응집도**: 0.78 (매우 높은 밀집도)
- **파일 수**: 20개의 Python 파일

## 4. 주요 기능

### 4.1 간 노드(God Node) 분석

가장 많이 연결된 핵심 허브 노드를 찾습니다.
- 예: `extract_graphly.py` — 48개 함수, 239개 관계 (핵심 허브)
- 새로운 프로젝트 온보딩 시 간 노드부터 시작하면 전체 구조 빠르게 파악

### 4.2 예상치 못한 연결(Surprising Connections)

간접적으로 연결된 모듈을 발견하여 숨겨진 의존성/암묵적 결합을 드러냄. 코드 리뷰·리팩토링에 유용.

### 4.3 추천 질문(Suggested Questions)

그래프 구조를 분석하여 탐색할 만한 질문 자동 생성:
- "extract와 build 사이의 데이터 흐름은?"
- "특허 출원과 등록의 차이는?"

### 4.4 Laydens 커뮤니티 감지

- 루베인(Leiden) 알고리즘 개선판
- 모듈러리티 최적화 + 연결 끊김 현상 해결
- `seed` 파라미터로 클러스터 크기 조절 가능
- 클러스터링 시각화: Color-coded node cluster visualization

## 5. 토큰 절감 효과

### 벤치마크

| 시나리오 | 그래프 없이 | GraphPy 사용 | 절감률 |
|---|---|---|---|
| 특허 법령 쿼리 | ~200,000 토큰 | 292 토큰 | **~685배** |
| 공식 벤치마크 | — | — | **5~71.5배** |

### 원리

1. **GraphPy 없음**: LLM이 전체 파일 읽음 (20만 토큰 이상)
2. **GraphPy 있음**: 미리 구축된 그래프만 참조 (292 토큰)
3. **초기 투자**: 그래프 생성 시 토큰 소모 (일회성)
4. **반복 쿼리**: 매번 그래프 재사용 → 비용 0

## 6. 성능 스펙

| 규모 | 처리 시간 |
|---|---|
| 50개 파일 (소규모) | < 1분 |
| 200개 파일 (보통) | < 5분 |
| 1,000개 파일 (대규모) | < 8분 |
| 10만 줄 코드 | ~8분 |
| 증분 업데이트 | 첫 실행 대비 **5배 이상** 빠름 |

### 성능 전략

1. **캐싱**: 해시 기반 → 변경 파일만 재처리
2. **청킹**: 파일 단위 청킹, 디렉토리 내 파일은 동일 청크에 배치
3. **보안**: 민감 파일(DMV, PEM, creds.json, AWS 키) 자동 필터링
4. **로컬 우선**: AST 추출 100% 로컬, 코드 외부 전송 없음

## 7. 기술 스택

| 구성 요소 | 기술 | 역할 |
|---|---|---|
| 코드 파싱 | **Tri-Sitter**(트리시터) | C 기반, 23개 언어, 밀리초 파싱, 오류 복원 |
| 그래프 생성 | **NetworkX** | 노드·엣지 추가, 경로 탐색, 중심성 계산 |
| 커뮤니티 감지 | **Laydens** (py-laydens) | 최적 커뮤니티 파티셔닝 |
| Python AST | **ast** 표준 라이브러리 | 고수준 보충 분석 (파이썬 전용) |
| 영상 변환 | **OpenAI Whisper** | 도메인 힌트 지원 전문 용어 인식 |

### 왜 트리티시터(트릭시터)인가?

| | Python AST | Tri-Sitter |
|---|---|---|
| 지원 언어 | Python 전용 | 23개+ 언어 |
| 속도 | 표준 | C 기반, 초고속 |
| 오류 복원 | × | ✓ (불완전 코드도 파싱) |
| 증분 파싱 | × | ✓ (변경 부분만 재파싱) |

## 8. 사용법

### 설치 (CLI 어시스턴트 활용)

```
Claude Code, Antigravity, Codex 등의 어시스턴트에 GitHub 주소를 제공하고
"이걸 스킬로 설치해줘"라고 요청하면 자동으로 설치됨
```

### CLI 명령어

```bash
# 현재 디렉토리 분석
graphpy

# 특정 폴더 분석
graphpy /path/to/folder

# Pass A만 (구조적 추출)
graphpy --pass-a

# GraphPy 쿼리 — 두 개념 간 경로 탐색
graphpy query "특허 출원과 등록의 차이"

# GraphPy 탐색 — BFS/DFS 기반 탐색
graphpy explore <node_name>

# GraphPy 설명 — 특정 노드에 대한 설명 생성
graphpy explain <node_name>

# GraphPy 위키 빌드 — Obsidian 호환 위키 생성
graphpy wiki

# GraphPy 감시 — 파일 변경시 자동 그래프 동기화
graphpy watch
```

### Output Output 폴더 구조 (`graphpy_out/`)

```
graphpy_out/
├── graph.json          # 쿼리 가능한 그래프 데이터
├── graph.html          # 인터랙티브 시각화 (D3.js)
├── graph.md            # 인사이트 보고서
├── cache/              # 해시 기반 캐시
└── ...
```

## 9. 실전 사용 사례

### 9.1 코드베이스 학습 (새 프로젝트 온보딩)

1. 프로젝트 폴더에 `graphpy` 실행
2. 간 노드로 핵심 모듈 파악
3. 연결 구조로 아키텍처 숨겨진 의도 이해
4. **온보딩 시간 → 크게 단축**

### 9.2 기술 부채 분석

- 과도하게 연결된 노드 → 기술 부채 징후
- 커뮤니티 분석으로 분리 가능한 모듈 발견
- 리팩토링 우선순위 결정

### 9.3 연구 코드베이스 관리

- 논문 + 구현 코드 함께 분석 → 이론과 구현 연결 확인
- 인용 관계 + 개념 통합 → 해당 분야 전체 지형도
- 개인 Raw Folder 관리 → 시간 지나도 쿼리 가능한 지식 저장소

### 9.4 대한민국 법령 분석 (Real-world 예제)

- **Legalize KR 프로젝트** (박정환 님): 판례 기반 대한민국 법령 관리
- 방대한 법령을 그래프로 시각화하여 LLM 질문에 정확한 답변 가능
- 특허법 분석 예제: 특허 출원→등록 전체 흐름 그래프

## 부록: AST(추상구문트리) 개론

- 소스 코드의 구조를 트리 형태로 표현한 자료 구조
- 코드 의미만 보존 (괄호, 세미콜론, 공백 등 구문 장饰은 제거)
- 언어 독립적 분석 가능 → 모든 언어를 동일 방식 분석
- 컴파일러/리터 기반 기술 → **결정론적** (확률적 아님)
- Python 표준 라이브러리 `ast` 모듈로 변환 가능
